# LLM 모의투자 연구소 개발 기획서

## 1. 프로젝트 목표

Mac mini에서 상시 실행되는 **LLM 기반 모의투자 연구소**를 구축한다.

이 프로젝트는 실제 자금을 투자하는 자동매매 시스템이 아니라, 여러 LLM이 동일한 시장 데이터를 보고 어떤 투자 판단을 내리는지 기록하고 비교하는 실험용 시스템이다.

초기에는 Gemini 펀드부터 구현하고, 이후 GPT 펀드와 Claude 펀드를 추가한다.

최종적으로 다음 항목을 비교할 수 있어야 한다.

* 모델별 투자 판단
* 모델별 포트폴리오
* 누적 수익률
* 일간 수익률
* 최대 낙폭
* 매수·매도 빈도
* 판단 근거
* 모델별 투자 성향 차이

---

## 2. 주요 구성

```text
NanoClaw + Claude
        │
        │ 비서 / 오케스트레이터
        ▼
Investment Core
        │
        ├─ 시장 데이터 수집
        ├─ 기술적 지표 계산
        ├─ LLM 투자 판단 요청
        ├─ 모의주문 실행
        ├─ 포트폴리오 계산
        └─ 보고서 생성
        │
        ▼
Supabase

외부 투자 판단 모델
├─ Gemini
├─ GPT
└─ Claude
```

### NanoClaw

NanoClaw는 전체 시스템의 비서이자 오케스트레이터로 사용한다.

주요 역할:

* 사용자와 대화
* 대화 세션 및 메모리 관리
* 예약 작업 실행
* Investment Core의 Tool 호출
* 투자 결과 조회
* 모델별 판단 비교
* 일간·주간 보고서 생성
* 자연어 요청을 내부 기능으로 연결

NanoClaw가 직접 포트폴리오 금액을 수정하거나 모의주문을 처리하지는 않는다.

### Claude

NanoClaw의 메인 두뇌는 Claude를 사용한다.

Claude는 투자 비교 실험에 참여하는 Claude 펀드와는 역할을 분리한다.

```text
Claude 오케스트레이터
= 시스템 관리, 사용자 대화, Tool 호출

Claude 투자 펀드
= 시장 데이터를 보고 투자 판단
```

두 역할은 같은 모델을 사용할 수 있지만 프롬프트, 세션, 책임을 분리한다.

## 3. 메모리와 데이터 저장

메모리는 두 영역으로 나눈다.

### NanoClaw 내부 메모리

다음과 같은 대화 및 운영 문맥을 저장한다.

* 이전 대화
* 사용자 요청
* 보고서 형식 선호
* 시스템 운영 규칙
* 채팅 세션
* 비서가 기억해야 할 프로젝트 정보

NanoClaw의 SQLite, 세션 파일, `CLAUDE.md` 및 워크스페이스는 Docker 재시작이나 Mac mini 재부팅 후에도 유지되도록 영속 볼륨을 사용한다.

### Supabase

Supabase는 투자 연구소의 공식 데이터 저장소로 사용한다.

다음 정보는 반드시 Supabase에 저장한다.

* LLM 에이전트 정보
* 투자 전략
* 시장 데이터 스냅샷
* 기술적 지표
* LLM 요청 및 응답
* 투자 판단
* 판단 근거
* 모의주문 및 체결
* 포트폴리오
* 보유 종목
* 현금 잔액
* 수익률
* 보고서
* 작업 실행 로그
* 오류 기록

NanoClaw 내부 메모리는 대화 연속성을 위한 것이고, Supabase는 투자 실험을 재현하고 비교하기 위한 저장소이다.

## 4. 투자 실험 구조

투자 모델과 투자 방식을 분리한다.

### 투자 모델

* Gemini
* GPT
* Claude

### 투자 방식

* ETF 투자
* 레버리지 ETF 투자
* 지수 투자

최종적으로 다음 9개 포트폴리오를 지원할 수 있어야 한다.

```text
Gemini × ETF
Gemini × 레버리지 ETF
Gemini × 지수

GPT × ETF
GPT × 레버리지 ETF
GPT × 지수

Claude × ETF
Claude × 레버리지 ETF
Claude × 지수
```

초기 MVP에서는 다음 하나부터 구현한다.

```text
Gemini × ETF
```

이후 Gemini의 레버리지 및 지수 포트폴리오를 추가하고, 마지막으로 GPT와 Claude를 동일한 인터페이스로 확장한다.

각 포트폴리오는 다음 상태를 독립적으로 가진다.

* 초기 투자금
* 현금
* 보유 종목
* 평균 매입가
* 실현 손익
* 미실현 손익
* 총 평가액
* 일간 수익률
* 누적 수익률
* 최대 낙폭

## 5. 공정한 모델 비교

각 모델은 반드시 동일한 조건에서 판단해야 한다.

* 동일한 시장 데이터
* 동일한 분석 시각
* 동일한 초기 투자금
* 동일한 투자 가능 종목
* 동일한 거래 수수료
* 동일한 슬리피지
* 동일한 위험 제한
* 동일한 응답 스키마
* 동일한 모의체결 방식

한 모델의 판단을 다른 모델에게 제공하지 않는다.

모델별 응답은 독립적으로 생성하며, 비교 보고서를 만들 때만 결과를 함께 조회한다.

## 6. 시장 데이터와 판단 방식

시장 데이터와 기술적 지표는 코드가 계산한다.

예시:

* 현재가
* 일간 등락률
* 5일·20일·60일 수익률
* 20일·60일·120일 이동평균
* RSI
* 변동성
* 거래량 변화
* 최근 고점 대비 낙폭
* 현재 포트폴리오 상태
* 최근 투자 판단

LLM은 계산을 직접 수행하기보다, 정리된 데이터를 보고 최종 투자 판단을 내린다.

초기에는 가격과 기술 지표 위주로 시작한다.

향후 다음 데이터를 추가할 수 있도록 확장 가능하게 설계한다.

* 뉴스
* 공시
* 금리
* 환율
* VIX
* 실적 발표
* 경제 일정

## 7. LLM 응답 형식

LLM 응답은 자유 텍스트가 아니라 구조화된 JSON으로 받는다.

예시:

```ts
interface InvestmentDecision {
  action: "BUY" | "SELL" | "HOLD" | "REBALANCE";

  orders: Array<{
    symbol: string;
    side: "BUY" | "SELL";
    targetWeight?: number;
    quantity?: number;
  }>;

  confidence: number;
  rationale: string[];
  riskFlags: string[];
  invalidationConditions: string[];
  portfolioComment: string;
}
```

LLM의 응답은 반드시 런타임 스키마 검증을 거친다.

다음 검증을 통과한 경우에만 모의주문을 실행한다.

* 허용된 종목인지
* 현금이 충분한지
* 보유 수량보다 많이 매도하지 않는지
* 최대 투자 비중을 넘지 않는지
* 같은 판단을 중복 실행하지 않는지
* 레버리지 제한을 넘지 않는지

LLM이 실패하거나 잘못된 응답을 반환하면 임의로 주문하지 않고 `HOLD` 또는 실행 실패로 기록한다.

## 8. 모의투자 엔진

실제 증권 주문 API는 연결하지 않는다.

Investment Core가 다음 기능을 담당한다.

* 모의주문 생성
* 모의체결
* 수수료 및 슬리피지 반영
* 현금 차감 및 증가
* 포지션 갱신
* 평균 매입가 계산
* 포트폴리오 평가
* 수익률 계산
* 중복 주문 방지

모든 주문은 재현 가능한 규칙으로 처리한다.

예시:

```text
미국 시장 마감 후 판단
→ 다음 거래일 시가에 모의체결
```

또는

```text
판단 당시 가격
→ 설정된 슬리피지를 반영하여 모의체결
```

사용한 체결 규칙은 설정과 DB에 기록한다.

## 9. toss_cmd와의 관계

`toss_cmd`는 이 프로젝트와 완전히 별개의 기존 프로젝트다.

LLM 투자 연구소는 `toss_cmd`가 실행되지 않아도 독립적으로 정상 동작해야 한다.

```text
llm-investment-lab
= 독립적인 투자 실험 서비스

toss_cmd
= 필요할 때 투자 연구소 데이터를 보여주는 외부 화면
```

두 프로젝트는 별도의 Git 저장소로 유지한다.

```text
~/projects/
├─ llm-investment-lab/
└─ toss_cmd/
```

코드를 직접 공유하거나 하나의 Monorepo로 합치지 않는다.

연결이 필요할 때만 HTTP API를 사용한다.

```text
toss_cmd
→ Investment Lab Query API
→ Supabase
```

`toss_cmd`는 다음과 같은 데이터를 조회해서 화면에 표시하는 역할만 담당한다.

* 모델별 포트폴리오
* 수익률 비교
* 최근 투자 판단
* 판단 근거
* 모의주문 기록
* 일간 및 주간 보고서

예시 API:

```http
GET /api/v1/lab/summary
GET /api/v1/portfolios
GET /api/v1/decisions
GET /api/v1/performance/comparison
GET /api/v1/reports/daily/latest
GET /api/v1/reports/weekly/latest
```

`toss_cmd`는 Supabase 테이블을 직접 조회하지 않고 Query API를 통해 데이터를 받는 것을 기본 원칙으로 한다.

연결되지 않은 상태에서도 두 프로젝트는 각각 정상적으로 실행되어야 한다.

## 10. 채팅 연결

초기에는 Discord를 사용하지 않는다.

NanoClaw가 지원하는 로컬 CLI 또는 최소한의 테스트 채널을 통해 기능을 확인한다.

향후 `toss_cmd`에 AI 채팅 패널을 추가할 경우 다음 구조로 연결한다.

```text
toss_cmd 채팅 패널
→ Agent Gateway
→ NanoClaw
→ Claude 오케스트레이터
→ Investment Core Tool
```

데이터 조회와 AI 채팅은 서로 다른 API로 분리한다.

```text
Query API
= 포트폴리오, 판단, 수익률, 보고서 조회

Agent Gateway
= NanoClaw 대화, 세션, 응답 스트리밍
```

`toss_cmd`에 채팅 기능을 연결하는 작업은 MVP 필수 사항이 아니라 향후 확장 단계로 둔다.

## 12. Docker 운영

Mac mini에서 장시간 실행할 예정이다.

권장 구성:

```text
Mac mini

NanoClaw
├─ Claude 오케스트레이터
├─ 세션
├─ 메모리
└─ 예약 작업

Docker Compose
├─ investment-core
├─ investment-worker
├─ query-api
└─ agent-gateway

External
├─ Supabase
├─ Gemini API
├─ OpenAI API
└─ Anthropic API
```

NanoClaw가 자체적으로 Docker 컨테이너를 제어하는 구조라면, NanoClaw 호스트를 무리하게 Docker Compose 안에 넣지 않는다.

먼저 공식 권장 방식으로 NanoClaw를 Mac mini에 실행하고, 투자 관련 서비스만 Docker Compose로 분리하는 방식을 검토한다.

모든 주요 서비스는 Mac mini 재부팅 후 자동으로 재실행되어야 한다.

## 15. MVP 완료 기준

다음이 동작하면 1차 MVP 완료로 본다.

1. Mac mini에서 시스템이 실행된다.
2. 투자 서비스가 Docker Compose로 실행된다.
3. NanoClaw와 Claude가 비서 역할을 수행한다.
4. NanoClaw 메모리가 재시작 후 유지된다.
5. Supabase에 투자 데이터가 저장된다.
6. Gemini × ETF 포트폴리오가 생성된다.
7. 정해진 시각에 시장 데이터를 수집한다.
8. Gemini가 구조화된 투자 판단을 반환한다.
9. 응답 검증 후 모의주문을 실행한다.
10. 현금, 포지션, 수익률을 계산한다.
11. NanoClaw에게 최근 판단과 판단 이유를 질문할 수 있다.
12. 일간 보고서를 생성할 수 있다.
13. 실제 주문 API는 호출하지 않는다.
14. `toss_cmd` 없이도 모든 기능이 정상 작동한다.
15. 설치 및 실행 방법이 README에 작성되어 있다.
